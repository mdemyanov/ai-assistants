# Адаптация AI-ассистента под разные роли

## CTO / Технический директор

### Специфика
- Много технических решений с долгосрочными последствиями
- Баланс между техническим долгом и новыми функциями
- Управление несколькими командами с разными технологиями

### Структура 02_DOMAIN

```
02_TECHNOLOGY/
├── platforms/       # Платформы компании
├── products/        # Продукты
├── services/        # Сервисы
├── architecture/    # Архитектурные решения
└── tech_radar/      # Технологический радар
```

### Рекомендуемые методологии
- DDD (Domain-Driven Design)
- ITIL
- Arc42

### Ключевые шаблоны
- ADR (Architecture Decision Record)
- Техническое review
- Инцидент postmortem

### Типичные ограничения
```markdown
- **Единая платформа** — все продукты на основной платформе
- **Core in-house** — ядро разрабатывается своими силами
- **Регуляторика** — соответствие требованиям ИБ
- **Нет дублирования** — один продукт для одной задачи
```

## CPO / Директор по продукту

### Специфика
- Приоритизация между многими инициативами
- Баланс между пользовательской ценностью и бизнес-метриками
- Координация с множеством стейкхолдеров

### Структура 02_DOMAIN

```
02_PRODUCTS/
├── products/        # Продукты и их состояние
├── roadmaps/        # Роадмапы
├── research/        # Исследования пользователей
└── metrics/         # Продуктовые метрики
```

### Рекомендуемые методологии
- Jobs to Be Done
- OKR
- RICE (приоритизация)
- Story Mapping

### Ключевые шаблоны
- Product Brief
- Feature spec
- Ретроспектива релиза

### Типичные ограничения
```markdown
- **Unit-экономика** — каждая фича должна окупаться
- **Платформенный подход** — общие компоненты для всех продуктов
- **Согласование с sales** — новые фичи согласовывать с продажами
```

## COO / Операционный директор

### Специфика
- Множество параллельных процессов
- Оптимизация и стандартизация
- Метрики эффективности

### Структура 02_DOMAIN

```
02_OPERATIONS/
├── processes/       # Описания процессов
├── metrics/         # Операционные метрики
├── projects/        # Внутренние проекты
└── vendors/         # Поставщики и партнёры
```

### Рекомендуемые методологии
- Lean
- Six Sigma
- ITIL

### Ключевые шаблоны
- Process description
- SLA agreement
- Vendor evaluation

### Типичные ограничения
```markdown
- **Бюджет** — в рамках утверждённого бюджета
- **Compliance** — соответствие регуляторным требованиям
- **SLA** — соблюдение уровней обслуживания
```

## HR Director / Директор по персоналу

### Специфика
- Много индивидуальных кейсов
- Конфиденциальность информации
- Долгосрочное развитие людей

### Структура 02_DOMAIN

```
02_HR_DOMAIN/
├── positions/       # Описания позиций
├── competencies/    # Модели компетенций
├── programs/        # HR-программы
└── analytics/       # HR-аналитика
```

### Расширение 07_PEOPLE
```
07_PEOPLE/
├── {имя}/
│   ├── profile.md       # Профиль (расширенный)
│   ├── 1-1/             # Встречи
│   ├── development/     # План развития
│   └── performance/     # Оценка эффективности
```

### Ключевые шаблоны
- Performance review
- Development plan
- Exit interview

### Типичные ограничения
```markdown
- **Конфиденциальность** — личные данные только для HR
- **Трудовое законодательство** — соответствие ТК
- **Политики компании** — единые правила для всех
```

## Руководитель проекта / PMO

### Специфика
- Множество проектов в разных статусах
- Управление рисками и зависимостями
- Отчётность для разных аудиторий

### Структура 02_DOMAIN

```
02_PROJECTS/
├── projects/        # Карточки проектов
├── risks/           # Реестр рисков
├── lessons/         # Lessons learned
└── reports/         # Шаблоны отчётов
```

### Рекомендуемые методологии
- PMBOK
- Agile (Scrum, Kanban)
- Critical Chain

### Ключевые шаблоны
- Project charter
- Status report
- Risk register
- Lessons learned

### Типичные ограничения
```markdown
- **Scope** — только в рамках согласованного scope
- **Budget** — строгий контроль бюджета
- **Timeline** — сроки согласованы со стейкхолдерами
```

## Универсальные элементы

### Структура 01_CONTEXT (для всех ролей)

```
01_CONTEXT/
├── role_scope.md      # Роль, полномочия, области ответственности
├── constraints.md     # "Красные линии" — что нельзя нарушать
├── stakeholders.md    # Ключевые стейкхолдеры и отношения
└── company.md         # Контекст компании (опционально)
```

### Структура 07_PEOPLE (для всех ролей)

```
07_PEOPLE/
├── {имя}/
│   ├── profile.md     # Профиль человека
│   └── 1-1/           # Записи встреч
```

### Обязательные шаблоны

| Шаблон | Назначение |
|--------|-----------|
| template_person.md | Профиль человека |
| template_1-1.md | Запись 1-1 встречи |
| template_decision.md | Фиксация решения (ADR) |
| template_task.md | Задача |

## Как адаптировать

### Шаг 1: Определить специфику роли
- Ключевые области ответственности
- Типы принимаемых решений
- Основные стейкхолдеры

### Шаг 2: Выбрать методологии
- Какие фреймворки используете?
- Какие метрики отслеживаете?

### Шаг 3: Определить ограничения
- Что категорически нельзя?
- Какие правила не нарушать?
- Что вне компетенции?

### Шаг 4: Адаптировать структуру
- Переименовать 02_DOMAIN под специфику
- Добавить специфичные шаблоны
- Настроить Dataview-запросы
