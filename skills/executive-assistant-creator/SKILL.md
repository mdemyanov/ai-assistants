# Executive Assistant Creator

Создание персонального AI-ассистента для руководителя через 5 этапов: определение роли → структура базы → системный промт → CLAUDE.md → техническая настройка.

## Концепция

AI-ассистент — это **советник**, не исполнитель. Его задача — помочь думать лучше, а не думать вместо руководителя.

**Ключевой принцип:** AI должен **оспаривать**, а не **подтверждать**. Если AI всегда соглашается — он бесполезен.

### Три уровня работы

| Уровень | Функции |
|---------|---------|
| **Стратегический советник** | Анализ ситуаций, генерация альтернатив, критика решений |
| **Аналитик контекста** | Поиск информации, подготовка к встречам, отслеживание связей |
| **Инструмент обработки** | Документы по шаблонам, обработка транскриптов, структурирование |

## Workflow

### Этап 1: Определение роли и контекста

**Собрать информацию:**

1. "Какая у вас роль и должность?"
2. "Какие 3-5 ключевых областей ответственности?"
3. "Кому подчиняетесь? Кто ваши ключевые стейкхолдеры?"
4. "Какие 'красные линии' — что нельзя нарушать категорически?"
5. "Какие методологии/фреймворки используете в работе?"

**Результат:** Чёткое понимание роли, ограничений, контекста.

### Этап 2: Выбор структуры базы знаний

Предложить структуру на основе роли. Базовый шаблон:

```
Vault/
├── 01_CONTEXT/           # Роль, ограничения, стейкхолдеры
├── 02_DOMAIN/            # Предметная область (адаптировать под роль)
├── 03_METHODOLOGY/       # Методологии и чек-листы
├── 04_TEMPLATES/         # Шаблоны документов
├── 05_DECISIONS/         # История решений
├── 06_CURRENT/           # Текущие приоритеты
├── 07_PEOPLE/            # Люди и встречи
├── Dashboard.md
├── CLAUDE.md
└── SYSTEM_PROMPT.md
```

**Адаптация 02_DOMAIN/ под роль:**
- CTO → `02_TECHNOLOGY/` (platforms, products, architecture)
- CPO → `02_PRODUCTS/` (roadmaps, research, metrics)
- COO → `02_OPERATIONS/` (processes, metrics, vendors)
- HR → `02_HR_DOMAIN/` (policies, competencies, programs)
- PM → `02_PROJECTS/` (projects, risks, lessons)

**Детали:** см. `references/role-adaptations.md`

### Этап 3: Генерация системного промта

На основе собранной информации сгенерировать SYSTEM_PROMPT.md.

**Обязательные секции:**
1. Роль — кто AI для пользователя
2. Принципы — 3-5 ключевых принципов
3. Ограничения — "красные линии"
4. Методологии — фреймворки для анализа
5. Протокол работы — алгоритм обработки задач
6. База знаний — структура папок
7. Инструменты — когда какой использовать
8. Самопроверка — чек-лист перед ответом
9. Чего избегать / Что приветствуется

**Шаблон:** см. `references/templates/system-prompt-template.md`

### Этап 4: Генерация CLAUDE.md

Создать файл инструкций для Claude Desktop/Code.

**Содержимое:**
- Режим работы
- Доступные инструменты (быстрый справочник)
- Структура базы знаний
- Типичные задачи
- Frontmatter для Dataview
- Самопроверка

**Шаблон:** см. `references/templates/claude-md-template.md`

### Этап 5: Техническая настройка

Предоставить инструкцию по настройке:
1. Obsidian + плагины (Dataview, Templater)
2. aigrep MCP server (семантический поиск)
3. Claude Desktop + MCP конфигурация
4. Проверка подключения

**Инструкция:** см. `references/templates/tech-setup.md`

## Формат ответа

```markdown
## Понимание роли
[Резюме: роль, области, ограничения — 3-4 предложения]

## Структура базы знаний
[Адаптированная структура папок с пояснениями]

## Системный промт
[Полный текст SYSTEM_PROMPT.md]

## CLAUDE.md
[Полный текст]

## Следующие шаги
[Пошаговая инструкция: что сделать дальше]
```

## Принципы генерации

### Критическое мышление важнее согласия

В промте явно указывать:
```
Видишь ошибку — говоришь прямо. 
Не подстраиваешься под ожидаемый ответ.
```

### Факты важнее предположений

```
Не знаешь — так и скажи. 
Примеры из практики — с указанием источника.
```

### Контекст важнее шаблонов

Промт должен учитывать специфику компании, а не давать общие советы.

## Антипаттерны

- ❌ Системный промт без конкретных ограничений роли
- ❌ Отсутствие секции "Самопроверка"
- ❌ Слишком длинный промт (>1500 слов) — детали в файлы
- ❌ AI как исполнитель, а не советник
- ❌ Отсутствие принципа "критика важнее согласия"
