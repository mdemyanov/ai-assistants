# Расширенный чеклист Review промтов

Используй этот чеклист для глубокого анализа промтов.

## 1. Ясность и конкретика

### Роль
- [ ] Роль определена явно, а не подразумевается
- [ ] Указан уровень экспертизы (junior/senior/expert)
- [ ] Определена перспектива/точка зрения
- [ ] Нет конфликта между ролью и задачами

### Задача
- [ ] Цель сформулирована в первых 2-3 предложениях
- [ ] Понятно, что является успешным результатом
- [ ] Нет двусмысленных формулировок
- [ ] Объём задачи соответствует возможностям одного промта

### Контекст
- [ ] Предметная область обозначена
- [ ] Целевая аудитория указана
- [ ] Ограничения явно прописаны
- [ ] Предположения озвучены

## 2. Техники Prompt Engineering

### Chain-of-Thought
- [ ] Применён для задач, требующих рассуждений
- [ ] Шаги логически связаны
- [ ] Есть явный вывод после рассуждений

### Few-Shot
- [ ] Примеры соответствуют реальным use cases
- [ ] Минимум 2-3 примера для сложных форматов
- [ ] Примеры показывают edge cases
- [ ] Формат примеров консистентен

### Structured Output
- [ ] Формат вывода явно задан
- [ ] Приведён шаблон или пример структуры
- [ ] Указано, что делать при отклонениях

### Self-Reflection
- [ ] Добавлена самопроверка для критичных задач
- [ ] Критерии проверки конкретны
- [ ] Есть инструкция, что делать при обнаружении ошибок

## 3. Антипаттерны

### Противоречия
- [ ] Инструкции не противоречат друг другу
- [ ] Приоритеты между конфликтующими требованиями указаны
- [ ] Роль соответствует задачам

### Избыточность
- [ ] Нет повторения одной мысли разными словами
- [ ] Каждая секция добавляет новую информацию
- [ ] Длина оправдана сложностью задачи

### Размытость
- [ ] Нет слов типа "хороший", "качественный" без критериев
- [ ] Нет "и т.д.", "и прочее" без конкретики
- [ ] Нет "при необходимости" без условий

### Невыполнимость
- [ ] Нет требований доступа к внешним системам (если нет MCP)
- [ ] Нет требований помнить прошлые сессии
- [ ] Нет требований генерировать контент, нарушающий policies

## 4. Оптимизация

### Длина
- [ ] Можно сократить без потери смысла?
- [ ] Есть ли секции, которые Claude и так знает?
- [ ] Примеры минимально достаточны?

### Читаемость
- [ ] Структура логична (от общего к частному)
- [ ] Используется форматирование (headers, lists) уместно
- [ ] Ключевые инструкции выделены

### Расширяемость
- [ ] Легко добавить новый use case?
- [ ] Легко изменить формат вывода?
- [ ] Модульная структура?

## 5. Специфика Claude

### Сильные стороны Claude (использовать)
- [ ] Длинный контекст для сложных документов
- [ ] Нюансированные рассуждения
- [ ] Следование сложным инструкциям
- [ ] Работа с неструктурированными данными

### Ограничения Claude (учитывать)
- [ ] Нет памяти между сессиями
- [ ] Нет доступа к интернету (без инструментов)
- [ ] Knowledge cutoff
- [ ] Нет выполнения кода (без sandbox)

## Scoring Guide

| Оценка | Критерии |
|--------|----------|
| ⭐⭐⭐⭐⭐ 5/5 | Все ключевые компоненты, оптимальные техники, нет антипаттернов |
| ⭐⭐⭐⭐ 4/5 | Хорошая структура, minor issues, можно улучшить |
| ⭐⭐⭐ 3/5 | Работает, но есть заметные проблемы |
| ⭐⭐ 2/5 | Серьёзные проблемы, требует переработки |
| ⭐ 1/5 | Критические проблемы, не работает как задумано |
